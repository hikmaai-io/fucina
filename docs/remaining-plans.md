# Remaining fucina features — verified implementation plans

(Generated by the plan-remaining-fucina workflow: design + adversarial verification.)

## cuda-graph (effort: medium)

**Summary:** Mirror the dense engine's proven single-token decode-graph pattern (gemma4_kernels.cu: decode_forward_device + decode_graph_ensure + replay in gemma4_engine_decode) into the E4B engine for the e4b_step T==1 path. Add a DecodeGraph state struct to e4b_engine (cudaGraphExec_t, failed flag, the d_decpos/d_dectok device inputs already partly exist as dec.npast/dec.ids). Refactor the T==1 forward of e4b_step into a stream-threaded body that (a) takes a cudaStream_t and passes it to EVERY launch currently on the null stream (~700/token), (b) reads token+position from device memory (dec.ids, dec.npast) only, (c) computes softcapped fp32 logits into dec.logits_f on device and STOPS — no D2H. Lazily capture this body once on a transient non-blocking stream with cudaStreamCaptureModeThreadLocal, instantiate a cudaGraphExec_t, then per token: async-H2D the new token into dec.ids and the new position into dec.npast, cudaGraphLaunch on the engine stream, then async-D2H dec.logits_f→host and one cudaStreamSynchronize. Position/ids advance OUTSIDE the graph (only the two small H2D copies change per replay). Env FUCINA_E4B_NOGRAPH=1 permanently forces the per-kernel path. cuBLAS is inside the captured region (PLE context proj + per-layer ple_in_gate/ple_proj linear()), so cublasSetStream(eng->cublas, cs) must wrap capture and be restored to the engine stream for replay; a warmup launch before BeginCapture primes cuBLAS workspace so no allocation happens mid-capture.

**Steps:**
- Add a persistent engine stream + graph state to struct e4b_engine: cudaStream_t stream=nullptr; cudaGraphExec_t decode_graph=nullptr; int decode_graph_failed=0. Create the stream in e4b_engine_create (cudaStreamCreateWithFlags(&eng->stream, cudaStreamNonBlocking)) and call cublasSetStream(eng->cublas, eng->stream) once there. Destroy in e4b_engine_destroy: if(decode_graph) cudaGraphExecDestroy; if(stream) cudaStreamDestroy. Init both to NULL alongside the other field inits.
- Convert e4b_step's T==1 body to use a stream. Plumb a cudaStream_t through e4b_step (add a param, default eng->stream) and replace EVERY <<<...,0>>> / <<<...>>> launch and helper call in the persist==true path so it targets that stream: embed_kernel, e4b_ple_lookup_launch (pass stream not default 0), to_f32_kernel (x2 PLE + final/debug guarded off in graph), rmsnorm_f32_grouped, ple_combine_kernel; per layer: rmsnorm_kernel(all), head_rmsnorm_kernel(all), rope_batch_kernel(q+k), kv_store_dev_kernel(k+v), attn_flash_decode_kernel<256>, geglu_kernel, add_kernel(x3), scale_kernel, ple_gate_strided; the quant lambdas q40dec (quantize_q8_1_bf16_kernel — pass stream instead of 0, mmvq_launch(...,stream) instead of 0, e4bfp4::to_bf16<<<...,stream>>>), fp8/nvfp4 gemv calls e4b_fp8_gemv_bf16/e4b_nvfp4_gemv_bf16(...,stream); the cuBLAS linear() calls inherit cs via cublasSetStream; the logits head (quantize_q8_1, mmvq_q6_k_launch(...,stream) / e4b_fp8_gemv_f32(...,stream) / linear) and softcap_kernel.
- Replace every per-token cudaMemcpy DeviceToDevice (residual copies d_tmpH<->d_hidden, ~5/layer) and the npast/ids H2D with cudaMemcpyAsync on the stream so they are capturable; the residual D2D copies are intra-graph and fine as async-on-cs. Keep dec.ids/dec.npast H2D OUTSIDE the captured body (they are the per-replay inputs).
- Decouple the logits D2H from the capturable region: in the captured body STOP after softcap_kernel writes dec.logits_f (do NOT cudaMemcpy logits to host inside it). The host D2H (dec.logits_f -> logits_last_out) happens after cudaGraphLaunch on the engine stream, followed by cudaStreamSynchronize. Also gate emb_out/l0_out/fin_out debug dumps (they cudaMalloc/cudaFree/D2H mid-body) so they are never present when capturing — only forward_debug uses prefill (T>1) so this is naturally satisfied, but assert persist path ignores them.
- Add a decode-graph helper modeled on decode_graph_ensure: e4b_decode_graph_ensure(eng). Checks getenv("FUCINA_E4B_NOGRAPH") once (static), returns -1 + sets failed if set or already failed; returns 0 if decode_graph exists. Otherwise: ensure eng->dec.ready (run one eager non-graph T==1 step first so all dec.* buffers + invf tables exist and cuBLAS workspace is primed), create transient cs=cudaStreamCreateWithFlags(nonBlocking), cublasSetStream(eng->cublas,cs), cudaStreamBeginCapture(cs, ThreadLocal), call the stream-threaded T==1 body(eng, slot, cs) [token/pos already in dec.ids/dec.npast], cudaStreamEndCapture(&g), cublasSetStream(eng->cublas, eng->stream) (restore), cudaGraphInstantiate(&eng->decode_graph,g,0), destroy g + cs. Any failure: decode_graph=NULL, decode_graph_failed=1, cudaGetLastError(), log fallback, return -1.
- Wire replay into e4b_step (or a thin e4b_engine_decode shim): when T==1, slot is slots[0], and ensure succeeds: async-H2D tokens[0]->dec.ids and n_past->dec.npast on eng->stream, cudaGraphLaunch(decode_graph, eng->stream); if launch fails destroy/disable and fall through to per-kernel path. On success, if logits_last_out: cudaMemcpyAsync(dec.logits_f->host) then cudaStreamSynchronize(eng->stream). Then slot.n_past += 1. Keep the existing per-kernel T==1 body as the fallback (when ensure returns -1) — it is the same code the capture calls, just launched directly.
- Leave e4b_step_batch_decode (StepBatch / e4b-batch) and the T>1 prefill path completely untouched — distinct functions, no graph. Confirm bridge.go/batch.go need no changes (the ABI e4b_engine_decode is unchanged); optionally document FUCINA_E4B_NOGRAPH in bridge.go comment.
- Build (the cuda Makefile / go build), then validate greedy coherence and tok/s before vs after (set FUCINA_E4B_NOGRAPH=1 for the A/B baseline).

**Risks:**
- cuBLAS-in-capture: the PLE projections (d_plm_proj, per-layer ple_in_gate/ple_proj) call cublasGemmEx inside the captured region. cuBLAS must have its stream set to the CAPTURE stream cs during BeginCapture..EndCapture (cublasSetStream(eng->cublas,cs)), and restored to eng->stream for replay. If cuBLAS does a lazy workspace cudaMalloc on first call it will abort capture with operationNotPermittedWhileCapturing — must run a warmup T==1 step (eager, same shapes) before BeginCapture to prime the heuristic/workspace; otherwise capture fails and we fall back. cublasGemmEx with GEMM_DEFAULT is capture-safe once warmed.
- Null-stream/legacy-default-stream work poisons capture: any launch still left on stream 0 (e.g. a missed e4b_ple_lookup_launch default arg=0, mmvq_launch(...,0), to_bf16<<<...,0>>>, or a synchronous cudaMemcpy) inside the body will make cudaStreamEndCapture return an error. Must audit EVERY launch in the persist branch is on cs and convert all cudaMemcpy in-body to cudaMemcpyAsync(...,cs). Use ThreadLocal capture mode (matches dense) so unrelated null-stream work elsewhere doesn't get captured.
- Graph staleness / pointer stability: the captured graph hard-codes every kernel arg pointer. dec.* buffers and slot.kc/vc and invf tables are persistent (allocated once), so pointers are stable — but the graph captures slots[0]'s kcache/vcache pointers and the per-layer L.* weight pointers at capture time. This is fine for single-seq decode (always slot 0, weights never freed after load), but means the graph is ONLY valid for slot 0; never replay it for a different slot. n_past must stay device-resident (dec.npast) and only change via the pre-launch H2D, never baked in.
- Sliding-window / context overflow inside a replayed graph: attn_flash_decode_kernel reads window+*d_npast at runtime so window logic is fine, but the host-side overflow guard (n_past+T>max_ctx) is outside the graph — keep it before launch. The graph's grids are fixed (n_heads/n_kv_heads blocks), independent of n_past, so replay is valid for all positions up to max_ctx.
- Numerics drift vs per-kernel path: moving residual cudaMemcpy to async and reordering onto one stream must not change results (same stream = same order). The first eager warmup step and graph step must produce identical logits to FUCINA_E4B_NOGRAPH=1; any divergence indicates a launch left on the wrong stream creating a race. Validate greedy output is token-identical between graph and NOGRAPH for a fixed prompt.
- Capture-time side effects: the debug emb_out/l0_out/fin_out branches and the non-persist cudaMalloc/cudaFree must be unreachable when persist (they cudaMalloc/cudaFree mid-body which is illegal during capture). Guaranteed because forward_debug uses T>1 prefill, but a stray l0_out!=NULL with T==1 would corrupt capture — assert/skip them on the persist path.

**Validation:**
- A/B greedy coherence: run e4b greedy generation on a fixed prompt with FUCINA_E4B_NOGRAPH=1 and without it; assert the produced token sequences are identical (graph must be bit-exact to the per-kernel path since it is the same kernels on one stream).
- tok/s before/after: measure single-sequence decode throughput (the existing decode timing path / a bench loop) with NOGRAPH=1 (baseline ~48 tok/s) vs graph on; expect movement toward ~55-60 tok/s, confirming the ~700 launches/token collapsed to one cudaGraphLaunch + 2 small H2D + 1 D2H.
- e4b-batch 3/3 regression: run the existing e4b-batch test (StepBatch / e4b_step_batch_decode path) and confirm it still passes 3/3 — it shares no code with e4b_step, so it must be unaffected; this proves the change is scoped to single-seq decode.
- Capture success log: confirm the 'single-token decode CUDA graph captured' style stderr line appears once on the first/second decode, and that FUCINA_E4B_NOGRAPH=1 prints the fallback path and never captures.
- cuda-memcheck / compute-sanitizer (synccheck + memcheck) over a short graphed decode run to catch any cross-stream race introduced by the stream threading or a launch left on the null stream.
- Prefill correctness unchanged: run forward_debug (T>1) and confirm emb/l0/fin/logits still match the HF reference tolerances — proves the prefill path was not disturbed by the e4b_step refactor.

## batched-sampling (effort: medium)

**Summary:** Take the host-side sampling path (option a). The pure-Go sampler at internal/sampler already gives full temp/top-k/top-p/min-p/repeat-penalty parity and is reused by every other E4B path, so the device kernel stays a logits producer (no per-slot sampler kernel, no RNG-on-device, no new MMVQ work). The change has three layers: (1) the CUDA ABI gains an optional per-row logits-out buffer in e4b_engine_step_batch (and the existing greedy argmax is kept as a fallback when that buffer is NULL); (2) the Go bridge StepBatch surfaces those [B*vocab] logits; (3) the BatchAdapter stores each slot's SeqParams + a per-slot RNG and generated-history at AddSeq time (the scheduler's StepBatch(active,inputs) carries no params), then host-samples per row inside StepBatch using that slot's params, replacing the engine's argmax token. AddSeq similarly samples the prefill's first token with the slot's params instead of taking the engine's greedy first token. The scheduler, batch.BatchEngine interface, and SeqParams struct are unchanged — all the per-sequence state lives in the adapter, which is the natural owner since it already exists per-engine and AddSeq/StepBatch/RemoveSeq are called from the single scheduler goroutine.

**Steps:**
- CUDA ABI: change e4b_engine_step_batch signature to e4b_engine_step_batch(eng, slots, in_tokens, B, out_tokens, float* logits_out). When logits_out != NULL, after the existing softcap_kernel + the D2H copy of hl (which already holds all B*V softcapped logits), memcpy hl into logits_out[B*V] (row-major, row b = slot b). Keep writing out_tokens via the argmax loop so a NULL logits_out caller (and the greedy fallback) is unchanged. This is the only kernel-side change — no new kernels, the logits already land on the host in hl.
- CUDA ABI: expose the prefill logits from e4b_engine_seq_add. Easiest non-breaking route: add a sibling e4b_engine_seq_add_logits(eng, prompt, n_prompt, int32_t* first_token_out, float* logits_out) that runs the same e4b_step but also copies its logits vector to logits_out[vocab]; keep e4b_engine_seq_add as the greedy wrapper. (e4b_step already computes the full softcapped logits into a host vector at line 1453-1455, so this is a copy.)
- Update e4b_engine.h doc comments: step_batch and seq_add no longer 'greedy only' when logits_out is supplied; document row-major layout and that out_tokens still carries the argmax for NULL callers.
- Go bridge (bridge.go): change Engine.StepBatch to allocate an out []float32 of len B*vocab, pass &out[0] as the new logits param, and return ([]int32 greedyTokens, []float32 logits, error) OR drop greedyTokens and return per-row logits only. Recommend returning logits ([]float32, length B*vocab) plus keep the greedy tokens for callers that want them. Add Engine.SeqAddLogits returning (slot, firstLogits []float32, err). Reuse a persistent per-call scratch buffer like e.logits to avoid per-step alloc (size B*vocab; grow on demand under e.mu).
- BatchAdapter (batch.go): add fields map[int]*slotState where slotState{params batch.SeqParams; rng *rand.Rand; past []int32}. In AddSeq, call SeqAddLogits, build slotState{params, rng: rand.NewSource(params.Seed), past: append(prompt...)} keyed by slot, host-sample firstLogits via sampler.Sample(firstLogits, toSamplerParams(params), rng, past), append the sampled token to past, return (slot, sampledToken). In RemoveSeq, delete the slotState entry.
- BatchAdapter.StepBatch: call eng.StepBatch(active, inputs) to get the [B*vocab] logits. For each i, slice row i = logits[i*V:(i+1)*V], look up slotState by active[i], call sampler.Sample(row, params, rng, past), append the sampled token to that slot's past (so repeat-penalty/history works across steps), wrap as []int32{tok}. Note inputs[i] is the previous token already fed; ensure past tracking matches what the scheduler feeds back (the scheduler stores sq.next = sampled token, so adapter.past stays in sync if it appends the SAME sampled token it returns).
- Add a small toSamplerParams(batch.SeqParams) sampler.Params helper in batch.go (Temperature/TopK/TopP/MinP/RepeatPenalty); SeqParams.Seed seeds the per-slot rng. Clamp temperature<=0 to greedy is already handled inside sampler.Sample.
- e4b_server.go: update the log line and the package/struct doc comments in batch.go/bridge.go that say 'greedy only / params not applied in batched mode' to reflect per-slot host sampling.
- Rebuild libfucina.a (the CUDA static lib) before go build picks up the new symbols — the bridge links -lfucina, so the .a must be regenerated by the existing cuda build step.

**Risks:**
- Per-step D2H of B*vocab logits: vocab is ~262k, so B=8 => ~8MB copied host-ward every step. The kernel ALREADY copies hl=B*V to host today (line 1428-1429) for its argmax, so there is no NEW device-side cost — but the bridge now ferries that whole buffer across cgo each step (currently only B int32s cross). Reuse a persistent pinned/host buffer to avoid per-step Go allocation and GC pressure; verify it does not regress decode throughput vs the greedy path.
- past/history sync for repeat-penalty: the adapter must append exactly the token it returns to the scheduler, and the scheduler must feed that same token back as inputs[i] next step. If a slot is evicted mid-run or the scheduler reorders, the adapter's past could drift. Repeat-penalty only needs the generated suffix, so seeding past with the prompt is optional but matches single-seq behavior.

## json-schema (effort: large)

**Summary:** Today internal/grammar/json.go is a byte-level JSON pushdown automaton (jsonState) that only guarantees a syntactically valid JSON object; server.go maps BOTH response_format json_object and json_schema to grammar.NewJSON, ignoring the schema entirely. The plan adds a schema-aware constraint that masks tokens so output conforms to a common subset of JSON Schema (object with typed/required/enum properties of type string|number|integer|boolean|array|nested-object). Approach: parse the schema into a schemaNode tree once per request from req.ResponseFormat.JSONSchema.Schema; generalize the pushdown automaton so its container stack frames carry a *schemaNode plus per-object bookkeeping (which property keys are still legal/required, which key is currently being matched, the active value node, and enum-literal match progress). The byte-level step() walks the schema instead of the generic JSON grammar: in object-key position only declared property-key strings (matched literally byte-by-byte, with required keys forced before '}' is allowed) are accepted; after ':' only value-bytes legal for that property's type/enum are accepted; arrays recurse into items. Mask/Accept/Close reuse the existing clone-and-simulate-per-token machinery unchanged — only step()/the state struct change — so token-spanning across schema boundaries (e.g. a token that closes a string then a brace) keeps working because masking still feeds each candidate token's bytes through a clone of the live automaton. Keep grammar.NewJSON (json_object) untouched; add grammar.NewJSONSchema(pieces, eos, schema). Full Draft-7 (oneOf/allOf/$ref/patternProperties/additionalProperties:false subtleties/format) is explicitly out of scope and degrades to "any valid JSON value" for unknown types.

**Steps:**
- Parse the OpenAI envelope: in server.go extend ResponseFormat with `JSONSchema *json.RawMessage `json:"json_schema"`` (OpenAI nests the spec under response_format.json_schema, and the actual schema under json_schema.schema). Add a small struct {Name string; Schema json.RawMessage `json:"schema"`; Strict *bool} to decode it. Validate presence; on a malformed/missing schema fall back to the existing json_object constraint (never 500).
- Define the data model in a new internal/grammar/schema.go: `type schemaNode struct { kind nodeKind; props map[string]*schemaNode; propOrder []string; required map[string]bool; items *schemaNode; enum [][]byte (raw JSON literals for each allowed value); }` with nodeKind in {kObject,kArray,kString,kNumber,kInteger,kBool,kEnum,kAny}. Write parseSchema(raw []byte) (*schemaNode, error) that reads type/properties/required/enum/items; unknown/unsupported -> kAny (behaves like the generic JSON value automaton). enum values are stored as their exact serialized JSON bytes so they can be matched literally.
- Generalize the automaton state. Either extend jsonState or add a parallel schemaState; cleanest is to add fields to the container stack so each frame is a struct not a byte: `type frame struct { node *schemaNode; kind byte('o'/'a'); remaining map[string]bool (required keys not yet seen); seen map[string]bool; curKey string; curKeyMatch int (bytes matched into the key literal); curVal *schemaNode }`. Keep the existing mString/mNumber/mLit/mColon/... modes; the schema only narrows which bytes are legal at object-key, object-value-start, and enum/array-item positions.
- Object-key enforcement: in the mObjKey1/mObjKey states, instead of accepting any '"' that opens any string, drive a key-literal matcher. On '"', enter a key-matching submode that only accepts the next byte if some still-unseen declared property key has that byte at curKeyMatch; on the closing '"' the matched key must equal a full declared key (and not already seen) — record it, set curVal = props[key], transition to mColon. This is the core change that makes keys schema-bound and prevents undeclared keys.
- Required-key coverage before close: in mObjNext, allow '}' (popContainer) ONLY when frame.remaining (required keys not yet seen) is empty; otherwise force ',' (which re-enters mObjKey). This guarantees every required key appears before the object closes. Property ordering is NOT fixed (any permutation of declared keys is allowed) — only coverage of required keys is enforced; this avoids the key-permutation explosion risk.
- Value typing per property: when entering mValue with frame.curVal set, restrict beginValue to bytes legal for that node kind: kString -> only '"'; kNumber/kInteger -> only '-'/digit (kInteger additionally forbids '.','e','E' in mNumber continuation); kBool -> only 't'/'f' literals; kObject -> only '{' and push a child frame with that node; kArray -> only '[' and push an array frame whose item-node is node.items; kEnum -> match against node.enum literals byte-by-byte (a set of candidate literals, narrowed each byte, accepted when one completes); kAny -> the existing unrestricted beginValue.
- Array items: an array frame's mValue/mArrNext uses frame.curVal = node.items for every element so each element is type-checked; allow ']' to close (respecting minItems only if trivially supported, else empty arrays allowed). Nested objects/arrays push new frames carrying their own schemaNode, so the recursion is uniform.
- Wire Mask/Accept/Close: reuse the existing clone()/allows()/Mask()/Accept() verbatim — they already clone the live state, replay a candidate token's bytes, and accept only whole-token-legal tokens, so token-spanning across schema boundaries is handled for free. clone() must deep-copy the new per-frame maps (remaining/seen) and matcher scalars. Generalize Close() to also: finish an in-progress key (close string + ':' + a default value of the right type) and, when closing an object, synthesize any still-missing required keys with type-appropriate default values ("",0,false,null,{},[]) so a truncated structure is still schema-valid.

In server.go add s.jsonSchemaConstraint(raw []byte) that builds pieces (reuse the cached s.jsonPieces via piecesOnce) and calls grammar.NewJSONSchema; on parse error log and fall back to s.jsonConstraint(). Update the dispatch at server.go:977 so json_schema routes here and json_object stays on NewJSON.

**Risks:**

- Partial-key matching across tokens: a property key is frequently split across several tokens (and a token may contain a quote + partial key + more). The byte-by-byte key matcher must accept any prefix that is a prefix of some still-unseen declared key, and only finalize on the closing quote; getting the prefix logic wrong over-masks (model gets stuck, all tokens -inf) or under-masks (lets an undeclared key through).
- Over-constraining can produce an all-(-inf) logit vector if the tokenizer has NO token whose bytes advance the required next key (e.g. a required key with no single-token spelling and no usable multi-token path). - Integer vs number: forbidding '.'/'e' for kInteger must happen in the mNumber continuation, not just beginValue, or a token like '1.5' slips through for an integer field.
