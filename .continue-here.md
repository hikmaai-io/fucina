## The four required reconciliations

### (a) The "slots=16" C32 confound — PRIMARY.md's framing was wrong; review's is right

PRIMARY.md claimed the 32→16 slot reduction was caused by shared-box memory pressure
("held by others" 44.47→77.69 GiB between the two fucina server starts). **This is
falsified by the review and I confirmed it myself**:

```
run 1 (fucina_server.log):  free=75.22 GiB, held-by-others=44.47 GiB, weights+scratch=57.86 GiB → slots=4/32
run 2 (fucina_server2.log): free=42.00 GiB, held-by-others=77.69 GiB, weights+scratch=25.64 GiB → slots=16/32
```
[RECONCILED, re-grepped from `benchmark-evidence/logs/fucina_server{,2}.log` directly]

Run 1 had **more** free memory and **less** external pressure, yet got **fewer** slots (4,
not 16) — the opposite of what PRIMARY.md's "external pressure caused it" story predicts.
I also missed that `fucina_server.log:24` already contains `slots=4/32` — PRIMARY.md
incorrectly said this wasn't logged. **Both errors are corrected here.**

The review's actual mechanism is right and better-evidenced: `ensure_q35_scratch()`
(`cuda/qwen35_runtime.cuh:170-187`) computes `per_slot` KV/state reservation sized to
**`maxctx`=25,280 tokens** (the worst case), not actual request length (17–5950 tokens in
this benchmark), then `fit = room / per_slot`. Because `per_slot` is large and fixed, `fit`
is extremely sensitive to `room` — small swings in available memory produce large swings in
granted slots (4 vs 16 vs a theoretical 32). This is a **documented, deliberate design
choice**, not a bug: `docs/gb10-inference-improvement-plan.md:97` states the acceptance
criterion explicitly — *"configured capacity is admitted only when its worst-case block
reservation fits; no request fails mid-decode because another slot consumed its promised
memory."* Line 14 of that same doc (dated 2026-07-09, one day before this benchmark, status
"active") names the exact issue: *"Admission must be based on allocations the engine can
guarantee, not a compile-time slot count or stale free-memory sample."* **fucina's own team
already identified this class of problem before this benchmark ran** — treat their
`Phase 1 (next)` in that doc as the primary spec, not a competing plan.

**What is still genuinely unresolved**: *why* `weights+scratch` itself differs 57.86 vs
25.64 GiB (a >2x swing) between two nominally identical launches of the same binary against
the same checkpoint with the same flags — both show `weight-cache ON (need 5.2 GiB, ...)`
identically, so that specific knob isn't the explanation [RECONCILED, re-checked both logs].
This is real, unexplained non-determinism and **must be root-caused before trusting any
capacity-planning fix** validated against a single run — see Step 1 below.

### (b) Grouped-MoE finding and "CUDA graphs don't explain 2k prefill" — full agreement, review adds precision

No disagreement here — both documents independently reached the same conclusion. The
review adds two things PRIMARY.md didn't have:
1. **Exact code-comment citations with numbers**, which I independently re-verified are
   real: `cuda/gemma4_kernels.cu:9279-9280` — *"measured 71.6% of a 2k-token prefill,
   ALU-bound at ~16 tok/expert"* — and `gemma4_kernels.cu:9329` — *"measured 314 ms per
   2.9k-token pass (11.6% of prefill)"* [RECONCILED, `grep` confirmed both strings present
   at those lines in the current tree]. Together >80% of prefill wall time, by the code's
   own prior self-measurement. This upgrades PRIMARY.md's "arithmetic estimate, ~30k
   launches" framing to a much stronger evidentiary basis — **but it is still a prior,
   self-reported number, not re-measured in this session**, on both documents' own
   admission. Live re-measurement (Step 1) still comes first.
2. **A cleaner mechanism citation for vLLM's side**: `moe_align_block_size()`
   (`vllm-src/vllm/model_executor/layers/fused_moe/moe_align_block_size.py:11-58`) sorts
   tokens into per-expert blocks device-side, then `fused_experts_impl`
   (`fused_moe.py:1556,1633`) runs as a single Triton kernel launch — no host round-trip, no
   per-expert loop, regardless of MoE backend (Triton, DeepGEMM, or CUTLASS all share this
   "few device-side launches regardless of expert count" property). This is a difference in
   *kind* (launch count scales with host-visible expert count for fucina; fixed/device-side
   for vLLM), not degree.

Both documents also agree, independently: chasing CUDA-graph capture for large prefill
shapes would **not** close this gap on either side (vLLM's own captured sizes cap at 512
tokens; a 2000+ token prefill runs through compiled-but-not-graphed code on vLLM too).
**Deprioritize CUDA-graph work for prefill entirely.**

### (c) Startup warmup vs per-request TTFT — PRIMARY.md's finding stands, unchallenged

PRIMARY.md (via the vLLM-cold-TTFT trace agent) established that vLLM front-loads ~175s of
torch.compile + FlashInfer autotune + CUDA-graph capture into server startup, entirely
before `Application startup complete` — invisible to the measured 807ms number. The
independent review does not address this angle and does not dispute it. **Preserved
as-is**: the 807ms vs 3506ms comparison is "cold-prompt-on-an-already-warm-process" for
both engines symmetrically, not "first request to a freshly launched process" for either —
a real production cold-start (autoscaling, spot preemption) would put vLLM's ~175s on the
critical path in a way fucina has no equivalent of. This matters for **prioritization**: the
MoE-kernel fix (Step 3below) targets steady-state uncached-prefix latency, not
deployment cold-start — a distinct, unaddressed axis worth naming explicitly to whoever
owns the roadmap, not a fix item in this plan.

### (d) When is async decode-scheduling actually justified — defer, and the review's own evidence reinforces deferring it

PRIMARY.md proposed async/pipelined decode scheduling (mirroring vLLM's `AsyncScheduler`,
which keeps sampled tokens GPU-resident across steps instead of fucina's `StepBatch`
round-tripping host-side token values every step) as a large, high-risk, "defer until
measurement justifies it" item (originally Phase 4). The review does not directly contest
this, but its own decode telemetry data **strengthens the case for deferring it**:
`fucina_server2.log:56,223` — engine cost per step goes from 16.78 ms at avgB≈1.0 to 55.69
ms at avgB=16.0, i.e. **per-sequence** cost drops ~16.8ms→~3.5ms, a ~4.8× per-sequence
efficiency gain from batching [RECONCILED, re-grepped and arithmetic re-checked]. **Decode
already scales well up to whatever slot ceiling it's allowed to reach.** The bottleneck at
C32 is admission/capacity (Steps 1-2 below), not decode-step synchronization overhead.
**Verdict: async scheduling stays a deferred, evidence-gated, last-resort item.** Do not
start it until Steps 1-4 are done AND a live profiler trace (not existing telemetry) shows
Go-scheduler-side per-step cost is competitive with GPU compute time at real (post-fix)
concurrency — which nothing measured so far indicates.

## Final recommended sequence

Ordered by dependency, not just priority — later steps assume earlier ones are done.
Every step's numbers must be re-measured against `benchmark-evidence/PROTOCOL.md`'s exact
protocol (same checkpoint, same diverse prompts, same temperature-0 sampling) before/after.

### Step 1 — Root-cause the weights+scratch/slot-count non-determinism — DONE
- Added an exact Qwen allocation ledger plus CUDA free-memory checkpoints, Linux
  `MemFree`/`MemAvailable`/page-cache telemetry, Q4_K/NVFP4 path decisions, MoE scratch
  accounting, weight-cache fit inputs and guaranteed-memory-plan inputs.
- Five sequential Qwen3.5-35B-A3B-FP8 restarts reproduced the split. The exact ledger stayed
  **23.13 GiB** every run. Warm observed load deltas were **24.77–25.79 GiB**; the first cold
  mmap pass was **47.79 GiB**, with a **24.45 GiB ledger residual**. Root cause: GB10 unified
  memory's `cudaMemGetInfo` follows raw free system pages, so populating the safetensors Linux
  file cache during a cold load was billed as CUDA “weights+scratch.” On warm runs those pages
  were already cached before the create-time baseline. This explains the historical
  57.86-vs-25.64 GiB swing; no model allocation path changed between starts.
- Allocation behavior is unchanged (diagnostic only). A later capacity-policy change may use
  reclaimable/available memory, but must retain the no-late-OOM guarantee.
- Gates observed: CUDA library + binary build, full `go test ./...`, and five live model starts.
- Steps 2–5 below are now executed as well.

### Step 2 — Live-instrument the prefill MoE path — DONE
- `--timings` now enables opt-in CUDA-event phase telemetry for Qwen MoE wide prefills:
  expert-slab dequant, router+route+gather, grouped experts, shared expert, and all remaining
  mixer/attention/head work. Normal serving has no events or synchronizations.
- Live warm 2013-token result on MoE-35B: **1376.87 ms total = 368.30 dequant + 51.45
  router/route + 372.03 grouped experts + 27.14 shared + 557.95 other**. The historical
  71.6% expert / 11.6% router prior is obsolete after grouped-batched GEMM; current grouped
  experts are 27.0% and router 3.7%.
- **Expected impact**: none directly — converts the 71.6%/11.6% prior comment (§b above)
  into a live, current-commit, current-prompt-length number, and directly validates whether
  Step 3 delivers what it should.
- **Effort/Risk**: LOW / LOW.
- **Acceptance criteria**: produces a reproducible ms-level breakdown of one cold
  ~2000-token prefill request that sums to ~100% of observed wall time.
- **Rollback**: n/a (instrumentation, env-var-gated).

### Step 3a — Retest `cublasGemmGroupedBatchedEx` — ALREADY SHIPPED, REVALIDATED
- Main already contains the grouped-batched cuBLAS path from `dc6c983`; the new phase telemetry
  confirms it is active and the 2k prefill is 1.38s warm in this shared-box run. MoE oracle
  parity remains 8/8 and batched row/graph independence remains 24/24 for all rows.
- **Expected impact**: HIGH if it now works (collapses the ~256-launches-per-projection
  loop to ~1 call, eliminating the per-layer host sync as a side effect) — **but this is
  not guaranteed and must not be assumed**; the fallback per-expert loop stays the default
  until this is proven safe.
- **Effort/Risk**: LOW effort / LOW risk **for the experiment itself** (isolated, easy to
  gate and abandon) — but flipping it to the default carries the same correctness risk as
  Step 3b below and must pass the identical gates.
- **Acceptance criteria**: bit-identical (or documented-tolerance) output vs the current
  per-expert-loop path across a **diverse-prompt** correctness suite (never identical-prompt
  — this exact stack has a documented history of identical-prompt benches hiding
  convergent-routing corruption). If it passes: proceed to make it default, skip Step 3b's
  new-kernel effort. If it fails: document the failure mode precisely (all-zeros vs
  nondeterministic vs something else) and move to 3b.
- **Rollback**: single env-var flip back to the existing per-expert loop.

### Step 3b — Replace the per-expert loop with a device-side grouped kernel — SKIPPED
- 3a passes and is already the default, so the contingency is unnecessary. The CUTLASS grouped
  NVFP4 path remains the decode path; wide prefill correctly uses grouped cuBLAS BF16.
- Historical contingency: reuse fucina's **already-working** CUTLASS grouped-GEMM primitive,
  `dg_fp4_moe_grouped` (`cuda/dg_fp4_moe.cu:139`), currently wired only for the NVFP4 quant
  path (`gemma4_kernels.cu:9345-9384`) — extend or adapt it to the FP8-block expert-weight
  case, replacing `gemm_bf16_grouped`'s host loop (`gemma4_kernels.cu:8014-8036`). This is a
  smaller lift than a green-field kernel because the dequant-to-BF16 step already exists
  (`gemma4_kernels.cu:9296-9320`) — the missing piece is only the GEMM dispatch.
  Also fold in the review's cheap, opportunistic item: batch the router GEMM across the
  full prompt instead of once per `GEMMA4_MOE_TMAX=2048`-token chunk (only helps prompts
  >2048 tokens, LOW effort, can ship independently/earlier).
- **Expected impact**: HIGH but sized by Step 2's live measurement, not the prior 71.6%/
  11.6% comment alone — target closing a meaningful (not necessarily 100%) fraction of the
  2.7s cold-TTFT gap.
- **Effort/Risk**: HIGH effort / MEDIUM-HIGH risk. MoE numerics on this exact GB10/CUDA13
  stack have a **documented history of silent corruption** (the very bug this whole code
  path exists to work around) — any new kernel needs the full existing correctness-gate
  bar, not a spot check.
- **Acceptance criteria**: (1) passes `make gpu-gates` and the `qwen35-*-test` Makefile
  targets (especially `qwen35-batch-test`, `qwen35-burst-test`, the MoE/FP8 oracle parity
  gates) at the same 8/8-or-better bar as today; (2) diverse-prompt bench, never
  identical-prompt; (3) measured cold-TTFT improvement via `turn_ttft.py` outside the ~2%
  run-to-run noise band (3446–3512 ms observed band).
- **Rollback**: gate behind an env var (matching the existing
  `g_fucina_force_slow_prefill`-style precedent) defaulting OFF until Step 3's acceptance
  criteria are met; single-flag revert thereafter.

### Step 4 — Decouple slot-count capacity planning from `maxctx` sizing — DONE
- Admission now reserves **8192 tokens/slot by default** (`FUCINA_QWEN35_SLOT_CTX` configures it;
  set it to maxctx for the old policy), while FULL KV remains geometric and may grow to maxctx.
  Growth checks exact owned bytes plus Linux `MemAvailable`; allocation stays transactional and
  declines before mutation if headroom disappears. This fixes the GB10 raw-`MemFree` page-cache
  confound without weakening CUDA allocation failure handling.
- The CLI concurrency ceiling is now the engine's real 32-row limit (was accidentally capped at
  16). Live result: **slots=32/32**, four concurrent 8513-token prompts crossed the 8K reservation
  successfully (4/4 HTTP 200), then a diverse C32 burst completed **32/32 HTTP 200, zero 503s**.
- **Expected impact**: HIGH — directly targets the measured 4-of-32 and 16-of-32 ceilings;
  this is the primary lever for the N=32 throughput plateau (not decode-kernel work — see
  reconciliation (d) above, decode already scales well).
- **Effort/Risk**: MEDIUM effort (mostly a formula + fallback-contract change, built on
  the already-shipped geometric per-slot KV growth, `qwen35_runtime.cuh:858-906`,
  `docs/gb10-inference-improvement-plan.md:94` marked done) / MEDIUM risk (must not
  reintroduce mid-generation starvation).
- **Acceptance criteria**: (1) a stress test that admits past the "typical" budget, grows
  several sequences past it simultaneously while other slots are also active, and confirms
  no OOM/crash/starvation; (2) increased slot count on the SAME benchmark protocol,
  ideally approaching the 32 ceiling on a quiescent box (Step 0-equivalent — re-run clean
  per the original PRIMARY.md Phase 0 recommendation, still valid and now a formal
  prerequisite of validating this step specifically); (3) no new HTTP 503s at the same
  `--max-concurrent 64` setting used in the original benchmark.
- **Rollback**: keep the old worst-case formula available behind a flag; this is a
  formula change, not a new code path, so revert is a small diff.

### Step 5 — Go-scheduler admission investigation — DONE; NO NEW POLICY NEEDED
- Main already includes post-analysis lockstep idle-burst coalescing (`3a415b2`): an idle short-
  prompt burst admits up to engine capacity in one pass, while a busy batch deliberately caps
  blocking one-shot admissions to one/pass to protect decode latency. Long prompts use the
  chunked/fused token-row budget. Adding a nominal token counter around sequential `AddSeq` would
  only increase active-decode stalls; it would not batch those independent prefills on GPU.
- Existing scheduler tests plus the live diverse C32 run validate the intended policy. Snapshot
  probing remains in the engine's per-request OpenSeq/AddSeq path; warm turn-2 reuse still works
  (live 2k conversation turn-2 TTFT 0.20s on the shared box, versus 1.62s cold).
- **Expected impact**: MEDIUM-HIGH, necessary complement to Step 4 — also directly
  addresses the TTFT-under-load growth (fucina's TTFT grows ~9× from N=1→16 even BEFORE the
  capacity ceiling is reached, because burst admission is sequential not parallel/batched).
- **Effort/Risk**: MEDIUM / LOW-MEDIUM (Go-side, more tractable than CUDA-side changes) —
  but see the explicit interaction risk below with the state-snapshot cache.
- **Acceptance criteria**: see the protection gates below (criterion 2 specifically).
- **Rollback**: Go-level change, straightforward single-PR revert.

### Explicitly deferred (do not start without new evidence)
- **Async/pipelined decode scheduling** (reconciliation (d)) — no current evidence
  justifies this; decode already scales well. Revisit only if a live profiler trace, taken
  AFTER Steps 1-5, shows Go-scheduler-side per-step cost competing with GPU compute time.
- **Genuine block-table paged KV for qwen35 FULL layers** — both documents agree this is
  the eventual "gold standard" (matches vLLM's design, matches fucina's own Phase 2), but
  HIGH effort/risk, touches the continuation-attention paths, needs full long-context +
  state-snapshot re-validation. Treat as a later phase, after Steps 1-5 are shipped and
  measured.
- **`maxctx` shared-memory ceiling fix** (tiled online-softmax, `docs/gb10-inference-improvement-plan.md`
  Phase 2.1/2.2) — raises the per-sequence context ceiling, indirectly shrinks Step 4's
  worst-case reservation too, but HIGH effort. Later phase.
- **`cublasGemmBatchedEx`** (true batched, device-pointer-array API) as a middle-ground
  between 3a and 3b — worth keeping in back pocket only if 3a fails AND 3b is deprioritized
  for effort reasons; do not pursue in parallel with 3b, pick one.

## A/B protection gates (adopted from the independent review's §11, unchanged — these are
## stronger/more precise than PRIMARY.md's original criteria and supersede them)

Every step above, before being considered mergeable:

1. **Bit-exactness gates stay green, no exceptions**: `make gpu-gates` plus the relevant
   `qwen35-*-test` targets (`qwen35-batch-test`, `qwen35-burst-test`, `qwen35-state-test`,
   MoE/FP8 oracle parity) at the same 8/8-or-better bar as today.
2. **Warm turn-2 TTFT must not regress by more than ~10%** from the measured 76.5ms median
   — this is fucina's clearest current win, produced by a different code path (state-
   snapshot cache) than any step above targets. **Named interaction risk**: Step 5's
   batched-admission work must not route snapshot-cache-eligible requests through a new
   batched path without first probing the cache per-request (`bridge.go:1401-1420`) —
   concrete mitigation: only batch confirmed cold-miss admissions together, never skip the
   per-request snapshot probe.
3. **Served tok/s at N=1,2,4,8,16 must not regress by more than ~5-8%** (matching the
   established ±8% noise band) from the measured baselines (32.2/58.2/101.7/154.6/206.0
   tok/s) — these are fucina's current wins over vLLM and are the explicit thing to
   protect, not just avoid catastrophic regression on.
4. **Cold turn-1 TTFT must measurably improve** from Step 3 — outside the ~2% run-to-run
   variance (3446-3512ms band observed).
5. **N=32 throughput is expected to improve from Step 4/5 but is not a hard gate** —
   partial progress is acceptable if 1-4 hold; the fix is staged, not a single change.
6. **No new HTTP 503s / admission failures** at the same `--max-concurrent 64` used in the
   original protocol.
